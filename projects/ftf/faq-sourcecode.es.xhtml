Esta FAQ contiene algunas informaciones más detalladas sobre los requerimientos y la mejor práctica corriente de poner a disposición el código fuente correspondiente para código ejecutable licenciado bajo GPL. Ha sido compilada como resultado de pequeños y numerosos defectos y faltas de más de secenta exitosas aplicaciones de licencia gpl.

Antes de hacer el lanzamiento (release) del código fuente como se require según la GPL, recomendamos que se lea esta FAQ para asegurar una satisfacción completa y mutua con el release que ud.está haciendo actualmente.

¿Qué tipo de código fuente debo publicar bajo los términos de GNU GPL?

La GNU GPL exige que si ud. distribuye software bajo licencia GPL en formato ejecutable también debe colocar en el dominio público "todo el código fuente correspondiente".Eso está definido del siguiente modo: "El código fuente de una obra constituye la forma preferida de trabajo para hacer modificaciones en ella. Para una obra ejecutable, el código fuente completo significa todo el código fuente de todos los módulos que contenga, además de cualquier archivo asociado que contenga definición de interfaces, además de los scripts usados para controlar la compilación y la instalación del ejecutable."

Esta es una definición muy precisa. Para un típico programa en C, esto se traduce en todo el código fuente (archivos .c) además archivos de cabecera (archivos .h )además de los scripts usadod para controlar la compilación y la instalación.

No olvide nunca que el objetivo de la GPL es permitir al usuario ejercer sus libertades. Una de estas libertades es la habilidad de hacer y correr versiones modificadas del porgrama.

¿Qué son los "scripts usados para controlar la compilación"?

Generalmente, cualquier programa sofisticado está dividido en diferentes archivos de código fuente. Durante el proceso de compilación, cada una de estos archivos es compilado en código ejecutable, luego de lo cual quedan enlazados unos a otros. Este proceso de compilación y de enlace individual de archivos de código es controlado generalmente con la ayuda de scripts. Con mucha frecuencia hay un programa que lo "hace", cuyos scripts son llamados por lo general como "Makefiles".

También, algunos programas tales como el núcleo (kernel) de Linux tienden a tener una configuración a tiempo de compilación (compile-time) (en el caso del kernel Linux el archivo .config). Dado que esta configuración a tiempo de compilación sin duda controla el proceso de compilación, ud. también necesita incluir todos los archivos de tal configuración a tiempo de compilación.

¿Qué son "scripts usados para controlar la instalación"?

Luego de haber traducido el software del código fuente a código ejecutable, el programa puede ser que tenga que ser instalado en un sistema. El proceso de instalación generalmente es automatizado por scripts de instalación. Estos son los que se mencionan en la GPL.

Por favor, nótese que esto es de especial importancia en caso de dispositivos embebidos que necesitan ser instalados para el programa ejecutable en el dispositivo. Si el usuario no ha recibido ningún modo de instalar su propia versión modificada del programa, no tiene manera de ejercer su libertad de correr versiones modificadas del programa.

A veces el proceso de instalación no es facilitado por scripts sino por otros métodos (tales como programas ejecutables). El texto de la GPL solo menciona la palabra "scripts". AL leer e interpretar la licencia, debe entenderse que la licencia no se refiere sólo y específicamente a "scripts", sino a cualquier tipo de programa de software que se requiera para instalar una version (modificada) del programa compilado.

¿Qué ocurre con el compilador, el toolchain?

La GPL declara explícitamente: "De todos modos, como un caso especial de excepción, el código fuente distribuido no necesita incluir nada que normalmente sea distribuido (sea en forma fuente o binaria) con los componentes mayores (compiladores, kernel, etc) del sistema operativo en el que corre el ejecutable, a no ser que la misma componente acompañe al ejecutable."

Este párrafo está escrito para programas de espacio de usuario que corren sobre sistemas operativos de propósito general. Si usted sólo distribuye tales programas, entonces no necesita incluir un compilador, un núcleo (kernel), un enlazador (linker), etc.

En caso de compilación cruzada (cross-compilation) para plataformas embebidas, el compilador cruzado para su hardware específico no es algo que sea normalmente distribuido con [...] el sistema operativo. Por tanto, ud. debe poner a disposición la versión exacta del compilador toolchain que fue usado para crear los ejecutables que ud. ha distribuido. Por favor, nótese que en la mayoría de los casos éste será el GNU gcc, el cuál está bajo la licencia GPL, de modo que las obligaciones de GPL quedarán aplicadas automáticamente al toolchain, y ud. también tendrá que hacer un release del código fuente completo que le corresponda.

¿De qué versión del código fuente debo hacer release?

Para cada una y cualquier versión del programa ejecutable, ud. debe hacer un release de la versión correspondiente del código fuente.

Así que si ud. ha distribuido diez versiones diferentes de Programación en Firme (firmware) para un producto embebido, y esta Programación en Firme contiene software de licencia GPL, entonces ud. tiene que hacer un release de diez diferentes paquetes de código fuente, en donde cada uno corresponda a cada versión individual del ejecutable.

Si elige la opción GPL 3b (en vez de la 3a), entonces tiene sólo la obliación de distribuirlo por máximo tres años. Significa que no necesita poner a disposición código fuente para cualquier código ejecutable que haya sido distribuido la última vez hace más de tres años.

No olvide que la distribución en medios de almacenamiento físico y vía red de datos tales como Internet está cubierto por la licencia GPL.

¿Cómo puedo verificar que el release de mi código fuente es completo?

Si usa código fuente puesto a disposición en un release específico, y ud. puede usar ese código fuente para producir una forma que trabaje del código ejecutable, entonces el release del código fuente parece estar completo.

Si el proceso de formación falla, o ud.termina con un ejecutable que no trabaja, o no ud. no encuentra modo de instalar el ejecutable resultante, entonces also está faltando.
